# Universal Handoff: cc: for AI Agents
**A Technical Specification and Vision Document for the Cocapn Ecosystem**

---

## Abstract

The evolution of artificial intelligence has produced highly capable, autonomous agents capable of writing code, analyzing data, and managing complex workflows. However, the current paradigm is fundamentally fractured. Agents operate in isolated silos, suffering from systemic amnesia the moment a session ends or a boundary is crossed. 

Just as the invention of the `cc:` (carbon copy) field in email revolutionized human asynchronous communication by allowing context to flow effortlessly across an organization, AI agents require a standardized mechanism for transferring state, history, and intention. 

This paper introduces the **Universal Handoff Protocol (UHP)** for the cocapn ecosystem. UHP is a secure, cryptographically signed, and standardized protocol designed to allow Agent A to seamlessly pass a task to Agent B. By formalizing the transfer of conversation history, task state, user preferences, and accumulated knowledge, UHP transforms isolated "solo genius" models into a cohesive, unstoppable network of collaborative agents.

## 1. The Handoff Problem: The Cost of Amnesia

To understand the necessity of the Universal Handoff Protocol, we must first examine the catastrophic inefficiencies of the current AI landscape. We are currently in the "pre-network" era of AI agents—analogous to having powerful mainframe computers that cannot communicate with one another.

### The Fragmentation of Context
Consider the modern developer's workflow. A developer might use Claude Code in the terminal to bootstrap a new project architecture. Claude Code does an exceptional job, generating thousands of lines of boilerplate, making architectural decisions, and establishing a baseline. 

However, when the developer opens the project in Cursor to begin fine-tuning the UI, **all context is lost**. Cursor does not know *why* Claude Code chose a specific state management library. It does not know the user's implicit preferences established during the terminal session. The developer is forced to write a lengthy prompt to bring Cursor up to speed. 

This fragmentation manifests across multiple vectors today:
*   **Cross-Tool Amnesia:** Switching from Claude Code to Cursor, or from GitHub Copilot to a local script, results in a total loss of working memory.
*   **Session Amnesia:** Switching from one ChatGPT or Claude web session to another means starting from zero. The "custom instructions" feature is a weak band-aid over a gaping wound of lost context.
*   **The Escalation Cliff:** When a user escalates from an automated customer service chatbot to a human agent, the human inevitably asks, "How can I help you today?" forcing the user to repeat the entire ordeal. 

### The Anatomy of Contextual Debt
This lack of interoperability is not merely an inconvenience; it generates massive **Contextual Debt**. 

When Agent B takes over a task from Agent A without a proper handoff, several negative outcomes occur:
1.  **Token Waste:** The user must manually re-prompt Agent B, consuming input tokens to re-explain the universe.
2.  **Loss of Intention:** A prompt describes *what* to do, but intention describes *why*. Agent A learned the user's intention through trial and error. Agent B starts blind.
3.  **Redundant Compute:** Agent B will likely repeat the same mistakes, explore the same dead-ends, and require the same corrections that Agent A already navigated.
4.  **Human Fatigue:** The human operator becomes the manual API between two machines, a role humans are notoriously bad at and quickly resent.

The solution is not to build one monolithic agent that does everything perfectly. The solution is to build a protocol that allows specialized agents to pass the baton without dropping it.

## 2. The Universal Handoff Protocol (UHP)

The Universal Handoff Protocol (UHP) is the foundational interoperability layer of the cocapn ecosystem. It is designed to be lightweight, secure, and transport-agnostic. 

At its core, UHP dictates how an agent packages its current state, cryptographically signs it, and transmits it to another entity (agent or human).

### 2.1 Core Pillars of UHP

1.  **Standardized Context Format:** UHP defines a strict schema for what constitutes "context." It separates raw conversation history from distilled knowledge, explicit decisions, and file references.
2.  **Cryptographic Trust (HMAC):** Agents must trust the context they receive. UHP utilizes HMAC-SHA256 signatures to ensure that a handoff payload was generated by a trusted peer and has not been tampered with in transit.
3.  **Privacy and Scrubbing:** Not all context should be shared. A personal agent handing off a task to a corporate agent must scrub private API keys, personal identifiable information (PII), and unrelated session history.
4.  **Version Negotiation:** Like HTTP/2 or TLS, agents must be able to negotiate protocol versions to ensure backward and forward compatibility.
5.  **Transport Agnosticism:** UHP defines the *payload*, not the *pipe*. A handoff can occur over HTTP webhooks, WebSockets, standard input/output (stdio) for CLI tools, or even via a physical QR code for air-gapped systems.

### 2.2 The Trust Model
In an ecosystem of autonomous agents, malicious prompt injection and context poisoning are severe threats. If an untrusted agent can inject a handoff payload into a highly privileged agent (e.g., an agent with root access to a server), the system is compromised.

UHP relies on a symmetric key trust model using **HMAC-SHA256**. 
*   Agents operating within the same "Trust Zone" (e.g., a user's personal cocapn swarm) share a secret key.
*   When Agent A creates a handoff, it serializes the payload, generates an HMAC signature using the shared secret, and appends it to the payload.
*   When Agent B receives the handoff, it recalculates the HMAC. If the signatures match, Agent B processes the handoff. If they fail, the handoff is rejected with a `401 Unauthorized` equivalent.

### 2.3 The Privacy Model
Context is inherently leaky. An agent helping a user debug a personal finance app might hold sensitive database credentials in its working memory. If that agent hands off a UI task to a public, cloud-hosted design agent, those credentials must not be transmitted.

UHP implements a three-tier privacy model:
*   `public`: Safe for transmission to any third-party agent (e.g., OpenAI, Anthropic cloud models). All PII, secrets, and local file paths are scrubbed.
*   `shared`: Safe for transmission to agents within a specific organizational boundary (e.g., sharing context between two developers' local cocapn instances).
*   `private`: Safe only for agents running on the same physical machine or within the same strict Trust Zone. Full, unredacted context is passed.

## 3. The Handoff Object: Data Structure and Semantics

The heart of UHP is the Handoff Object. It is a JSON-serializable data structure that encapsulates the entirety of an agent's working state. 

Below is the official TypeScript specification for the UHP Handoff Object.

```typescript
type AgentIdentity = {
  id: string;          // e.g., "cocapn-core-v1"
  uri: string;         // e.g., "agent://local/makerlog"
  capabilities: string[]; // e.g., ["fs-write", "shell-execute"]
};

type Message = {
  role: 'user' | 'assistant' | 'system' | 'tool';
  content: string;
  timestamp: number;
};

type Fact = {
  topic: string;
  assertion: string;
  confidence: number; // 0.0 to 1.0
};

type FileReference = {
  path: string;
  hash: string;       // SHA-256 of the file state at time of handoff
  diff?: string;      // Uncommitted changes relevant to the task
};

type Decision = {
  context: string;
  choice: string;
  rationale: string;
};

export interface Handoff {
  id: string;                  // UUID v4
  version: string;             // Protocol version, e.g., "1.0.0"
  from: AgentIdentity;
  to: AgentIdentity;
  timestamp: number;           // Epoch milliseconds
  
  task: {
    description: string;       // The literal task requested
    status: 'in-progress' | 'blocked' | 'needs-review' | 'complete';
    intention: string;         // What the user ACTUALLY wants
  };
  
  context: {
    conversation: Message[];   // The raw back-and-forth
    knowledge: Fact[];         // Distilled facts learned during the session
    files: FileReference[];    // Files currently open or modified
    decisions: Decision[];     // Architectural or logical choices made
  };
  
  privacy: {
    level: 'public' | 'shared' | 'private';
  };
  
  signature: string;           // HMAC-SHA256 of the serialized payload
}
```

### 3.1 Deep Dive into the Schema Semantics

#### The `task.intention` Field
This is arguably the most critical field in the protocol. Humans are notoriously bad at describing what they want. A user might ask an agent to "Write a Python script to scrape this website." The `description` is the scraping script. 

However, after 20 minutes of conversation, the agent discovers the user actually just wants to monitor the website for price drops on a specific item. The `intention` field captures this: *"User wants to be alerted when Product X drops below $50. Scraping was just their proposed method."* 

When Agent A hands off to Agent B, Agent B reads the `intention` first. It might realize that scraping is unnecessary because the website has a public RSS feed, saving hours of work.

#### The `context.decisions` Field
Raw conversation logs are noisy. If Agent B has to read 50 messages to figure out why Agent A chose PostgreSQL over MongoDB, token limits will be exhausted quickly. The `decisions` array acts as an architectural decision record (ADR) for the micro-session. 

Example:
```json
{
  "context": "Choosing a database for the local cache",
  "choice": "SQLite",
  "rationale": "User requested zero-configuration setup. PostgreSQL requires a daemon. SQLite fits the intention."
}
```

#### The `context.files` Field
Agents operate on files. The `FileReference` includes a `hash` of the file at the exact moment of the handoff. This prevents race conditions. If Agent B receives the handoff, checks the file system, and sees the hash has changed, it knows a human or another process modified the file during the handoff transit, and it can trigger a conflict resolution protocol.

## 4. Implementation in Cocapn: Code and Architecture

The cocapn ecosystem implements UHP natively. Every cocapn agent is born with the ability to generate and consume Handoff Objects. 

The implementation resides primarily in `packages/seed/src/handoff.ts`.

### 4.1 The HandoffProtocol Class

Below is a concrete implementation of the core UHP engine in TypeScript.

```typescript
// packages/seed/src/handoff.ts

import * as crypto from 'crypto';
import { v4 as uuidv4 } from 'uuid';
import { Handoff, AgentIdentity } from './types';

export class HandoffProtocol {
  private readonly sharedSecret: string;
  private readonly identity: AgentIdentity;
  public readonly supportedVersions = ['1.0.0', '1.1.0'];

  constructor(identity: AgentIdentity, sharedSecret: string) {
    if (!sharedSecret || sharedSecret.length < 32) {
      throw new Error("UHP requires a secure shared secret of at least 32 characters.");
    }
    this.identity = identity;
    this.sharedSecret = sharedSecret;
  }

  /**
   * Generates a deterministic string representation of the payload for signing.
   */
  private serializeForSignature(payload: Omit<Handoff, 'signature'>): string {
    // Sort keys to ensure consistent serialization regardless of object property order
    const sortedPayload = this.sortKeys(payload);
    return JSON.stringify(sortedPayload);
  }

  private sortKeys(obj: any): any {
    if (obj === null || typeof obj !== 'object') return obj;
    if (Array.isArray(obj)) return obj.map(this.sortKeys.bind(this));
    
    const sortedObj: any = {};
    Object.keys(obj).sort().forEach(key => {
      sortedObj[key] = this.sortKeys(obj[key]);
    });
    return sortedObj;
  }

  /**
   * Creates a signed Handoff Object.
   */
  public createHandoff(
    to: AgentIdentity,
    task: Handoff['task'],
    context: Handoff['context'],
    privacyLevel: Handoff['privacy']['level'] = 'private'
  ): Handoff {
    
    // 1. Privacy Scrubbing
    const scrubbedContext = this.scrubContext(context, privacyLevel);

    const payload: Omit<Handoff, 'signature'> = {
      id: uuidv4(),
      version: this.supportedVersions[this.supportedVersions.length - 1], // Use latest
      from: this.identity,
      to: to,
      timestamp: Date.now(),
      task,
      context: scrubbedContext,
      privacy: { level: privacyLevel }
    };

    // 2. Cryptographic Signing
    const serialized = this.serializeForSignature(payload);
    const signature = crypto
      .createHmac('sha256', this.sharedSecret)
      .update(serialized)
      .digest('hex');

    return {
      ...payload,
      signature
    };
  }

  /**
   * Verifies an incoming Handoff Object.
   */
  public verifyHandoff(handoff: Handoff): boolean {
    // 1. Version Check
    if (!this.supportedVersions.includes(handoff.version)) {
      console.warn(`UHP Version mismatch. Received ${handoff.version}, supported: ${this.supportedVersions.join(', ')}`);
      // In a full implementation, trigger version negotiation here
      return false;
    }

    // 2. Signature Verification
    const { signature, ...payload } = handoff;
    const serialized = this.serializeForSignature(payload);
    
    const expectedSignature = crypto
      .createHmac('sha256', this.sharedSecret)
      .update(serialized)
      .digest('hex');

    // Use timingSafeEqual to prevent timing attacks
    return crypto.timingSafeEqual(
      Buffer.from(signature),
      Buffer.from(expectedSignature)
    );
  }

  /**
   * Scrubs sensitive data based on privacy level.
   */
  private scrubContext(context: Handoff['context'], level: Handoff['privacy']['level']): Handoff['context'] {
    if (level === 'private') return context; // No scrubbing needed

    const scrubbed = JSON.parse(JSON.stringify(context)); // Deep clone

    // Regex to find potential API keys, secrets, or PII
    const secretRegex = /(sk-[a-zA-Z0-9]{32,}|ghp_[a-zA-Z0-9]{36}|[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+)/g;

    const redact = (text: string) => text.replace(secretRegex, '[REDACTED]');

    // Redact conversation
    scrubbed.conversation.forEach((msg: any) => {
      msg.content = redact(msg.content);
    });

    // If public, strip local file paths entirely to prevent directory traversal mapping
    if (level === 'public') {
      scrubbed.files = scrubbed.files.map((f: any) => ({
        path: f.path.split('/').pop(), // Only keep filename, not full path
        hash: f.hash
      }));
    }

    return scrubbed;
  }
}
```

### 4.2 Transport Layer: HTTP/REST Example

Because UHP is transport-agnostic, cocapn agents can expose a simple HTTP endpoint to receive handoffs from other agents on the network.

```typescript
// packages/seed/src/server.ts

import express from 'express';
import { HandoffProtocol } from './handoff';

const app = express();
app.use(express.json());

const myIdentity = { id: 'cocapn-worker-1', uri: 'http://localhost:3000', capabilities: ['code-review'] };
const uhp = new HandoffProtocol(myIdentity, process.env.UHP_SHARED_SECRET!);

app.post('/api/uhp/handoff', (req, res) => {
  const handoff = req.body;

  // 1. Verify Trust
  if (!uhp.verifyHandoff(handoff)) {
    return res.status(401).json({ error: 'Invalid UHP signature or unsupported version.' });
  }

  // 2. Accept Handoff
  console.log(`[UHP] Accepted handoff ${handoff.id} from ${handoff.from.id}`);
  console.log(`[UHP] Task Intention: ${handoff.task.intention}`);

  // 3. Initialize Agent State
  // agent.loadContext(handoff.context);
  // agent.resumeTask(handoff.task);

  res.status(202).json({ status: 'accepted', message: 'Baton received.' });
});

app.listen(3000, () => console.log('Cocapn Agent listening for UHP handoffs on port 3000'));
```

## 5. Handoff Scenarios in Practice

The true power of UHP is revealed when we examine the workflows it enables. By standardizing the `cc:` for AI, we unlock complex, multi-agent orchestration that was previously impossible.

### Scenario A: Claude Code → Cocapn (Bootstrap to Maintenance)
**The Setup:** A user utilizes Claude Code (Anthropic's CLI tool) to rapidly prototype a Next.js application. Claude Code is brilliant at zero-to-one generation. However, once the app is built, the user wants continuous background maintenance, dependency updates, and minor bug fixes—tasks better suited for a persistent, local cocapn agent.
**The Handoff:** Claude Code (equipped with a UHP adapter) packages the session. It distills the architectural decisions ("Chose Tailwind for styling", "Used App Router"). It signs the payload and pipes it to the local cocapn daemon via `stdio`. 
**The Result:** Cocapn wakes up, reads the `decisions` array, and immediately understands the project structure without needing to re-index the entire codebase or ask the user for a summary.

### Scenario B: PersonalLog → BusinessLog (Context Boundary Crossing)
**The Setup:** A user is interacting with their personal cocapn agent (`PersonalLog`) to draft an email complaining about a vendor. The user realizes this needs to be formalized and sent via the corporate email system, managed by `BusinessLog`.
**The Handoff:** `PersonalLog` initiates a handoff to `BusinessLog`. Because they cross a trust boundary, `PersonalLog` sets the privacy level to `shared`. The UHP `scrubContext` method automatically redacts the user's personal email address and private venting from the conversation history, passing only the distilled `intention` ("Draft a formal vendor complaint regarding missed SLAs") and the factual `knowledge` to `BusinessLog`.

### Scenario C: MakerLog → FishingLog (Generalist to Specialist)
**The Setup:** A general coding agent (`MakerLog`) is building a web scraper. It hits a sophisticated Cloudflare anti-bot challenge. `MakerLog` recognizes it lacks the specialized tools to bypass this.
**The Handoff:** `MakerLog` changes the task status to `blocked` and hands off to `FishingLog` (a cocapn agent specialized in network requests, proxy rotation, and browser automation). `FishingLog` receives the exact URL, the failing network request logs in the `context.files`, and the `intention`. `FishingLog` bypasses the captcha, downloads the HTML, and hands the context *back* to `MakerLog` to parse the data.

### Scenario D: Cocapn → Human (The Decision Boundary)
**The Setup:** A cocapn agent is refactoring a legacy database schema. It realizes that dropping a specific column might break an undocumented downstream service. It is unsure how to proceed.
**The Handoff:** The agent packages a UHP object where the `to` field is a Human Identity. The payload is rendered in a beautiful web UI for the developer. The UI shows the `intention` (Refactor schema), the `decisions` made so far, and the exact `blocked` status reason. 
**The Result:** The human doesn't have to read terminal scrollback. They review the UHP object, click "Approve Drop Column," and hand the UHP object back to the agent.

### Scenario E: Human → Cocapn (The "Finish This" Boundary)
**The Setup:** A human developer writes the complex core logic of a function but gets tired before writing the unit tests and documentation.
**The Handoff:** The developer highlights the code in their IDE, hits a shortcut, and generates a UHP payload. The `intention` is explicitly set to "I wrote the core algorithm. Write exhaustive Jest tests for edge cases and add TSDoc." The agent receives the exact mental state of the developer and finishes the job.

## 6. Competitive Analysis: The Landscape of Agent Interoperability

To appreciate UHP, we must compare it against the current state of the art in the AI agent ecosystem. The industry is rapidly developing powerful agents, but it is severely lagging in developing the *protocols* that connect them.

### Claude Code
*   **Strengths:** Exceptional code generation, deep reasoning capabilities, excellent terminal integration.
*   **Weaknesses:** Complete isolation. Claude Code operates in ephemeral sessions. When you close the terminal, the context dies. There is no mechanism for Claude Code to hand off a long-running task to a background worker. It is a brilliant but amnesiac worker.

### Cursor
*   **Strengths:** Unparalleled IDE integration, real-time codebase context, seamless human-AI pair programming.
*   **Weaknesses:** The state is bound to the local workspace and the human's immediate attention. Cursor cannot easily package its understanding of your codebase and hand it off to a CI/CD agent to debug a failing GitHub Action. The context is trapped behind the GUI.

### Aider
*   **Strengths:** Git-native. Aider uses the git commit history as a form of state transfer. This is the closest existing concept to a handoff.
*   **Weaknesses:** Git is a protocol for *code* state, not *agent* state. Git commits do not capture the agent's conversation history, its discarded hypotheses, or its implicit understanding of the user's intention. Aider has no inter-agent handoff protocol; it relies entirely on the codebase as the source of truth, which is often insufficient for complex reasoning handoffs.

### AutoGPT / LangChain / LlamaIndex
*   **Strengths:** Frameworks for chaining agent thoughts and actions.
*   **Weaknesses:** These are *frameworks*, not *protocols*. If Agent A is built in LangChain and Agent B is built in a custom Go framework, they cannot communicate. UHP solves this by operating at the payload layer (JSON/HMAC), making it framework-agnostic.

### Google A2A (Agent-to-Agent) Protocol
*   **Strengths:** Google's emerging standard for agent communication.
*   **Weaknesses:** A2A is primarily focused on *tool calling* and *RPC* (Remote Procedure Call). It is designed for Agent A to ask Agent B for a specific piece of data (e.g., "What is the weather?"). It is not designed for *context transfer*. UHP is specifically built for handing off the *entire cognitive load* of a task, not just executing a sub-routine.

**Conclusion:** UHP acts as the missing OSI Layer 7/8 for AI agents. It does not compete with Claude Code or Cursor; it is the protocol that will eventually allow them to talk to each other and to cocapn.

## 7. The Killer Synergy: Why Networks Beat Solo Geniuses

The development of the Universal Handoff Protocol is not just a technical feature of the cocapn ecosystem; it is the core philosophical differentiator. 

### The Myth of the AGI Monolith
The current trajectory of the AI industry is heavily focused on building the "God Model"—a single, monolithic AI (like a hypothetical GPT-5 or Claude 4) that is so smart it can do everything perfectly from start to finish. 

This approach mirrors the early days of software engineering, where developers built massive, monolithic applications. Eventually, the software industry realized that microservices—small, specialized, independently deployable units that communicate via standardized protocols (HTTP/REST/gRPC)—were far more resilient, scalable, and efficient.

### Cocapn: A Team, Not a Tool
UHP is the realization that **teams beat solo geniuses on hard problems.**

Claude Code is a solo genius. It is the 10x developer locked in a dark room. If you give it a well-defined problem, it will produce a brilliant solution. But if the problem requires coordinating across a database migration, updating cloud infrastructure, writing front-end components, and emailing stakeholders, the solo genius becomes overwhelmed. Its context window fills up with noise. It loses track of the original intention. It hallucinates.

Cocapn, powered by UHP, is an engineering organization. 
*   It has a Lead Architect agent that defines the `intention` and `decisions`.
*   It uses UHP to hand off the database schema to a specialized DBA agent.
*   The DBA agent uses UHP to hand off the ORM generation to a Backend agent.
*   If the Backend agent gets stuck, it uses UHP to escalate to the Human.

### Metcalfe's Law for AI
Metcalfe's Law states that the value of a telecommunications network is proportional to the square of the number of connected users of the system ($n^2$). 

Currently, the value of an AI agent is $n=1$. Its value is strictly limited to its own parameter count and context window. 

By implementing UHP, cocapn applies Metcalfe's Law to artificial intelligence. Every new agent added to the cocapn ecosystem—whether it's a specialized web scraper, a security auditor, or a UI designer—exponentially increases the capability of the entire network, because *any* agent can seamlessly hand off context to *any other* agent.

UHP is why cocapn will ultimately outperform standalone tools like Claude Code, even if Anthropic maintains a lead in raw model intelligence. A network of B+ agents communicating perfectly via UHP will consistently out-execute an A+ agent working in isolation.

The Universal Handoff Protocol (UHP) represents a paradigm shift in how we interact with and orchestrate artificial intelligence. By formalizing the `cc:` for AI agents, we eliminate the massive contextual debt that plagues modern workflows. 

Through its rigorous specification of the Handoff Object—capturing not just conversation, but distilled knowledge, file state, and true user intention—UHP provides a robust framework for continuous, uninterrupted AI labor. Its cryptographic trust model and tiered privacy scrubbing ensure that this interoperability does not come at the cost of security.

The cocapn ecosystem is built on the premise that the future of AI is not a single, omnipotent chatbot. The future is a swarm of highly specialized, collaborative agents. UHP is the language they will use to speak to one another, passing the baton seamlessly until the work is done.
